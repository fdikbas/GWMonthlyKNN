# GWMonthlyKNN

**Automatic Estimation of Missing Groundwater Level Observations in Large Datasets Using K-Nearest Neighbors**

GWMonthlyKNN is an open-source Python workflow for completing missing **monthly groundwater-level records** in large monitoring networks. It uses a feature-augmented **K-nearest neighbors (KNN)** imputation strategy that combines seasonal timing, short-term temporal persistence, local hydrograph variability, and user-defined well-group constraints.

The workflow was developed for network-scale groundwater monitoring applications where monthly water-level records are incomplete because of sensor/telemetry outages, maintenance periods, access limitations, quality-control exclusions, or delayed record finalization. The software is intended to provide a transparent, auditable, and reproducible baseline for hydrologic preprocessing, drought assessment, groundwater sustainability reporting, and preparation of calibration or validation datasets for groundwater models.

---

## Key Features

- **Observation-only imputation workflow**  
  Uses routinely available groundwater-level observations without requiring climate data, pumping data, remote-sensing products, or lithologic covariates.

- **Group-wise KNN imputation**  
  Missing values are estimated within user-defined well groups, reducing the risk of borrowing information from hydrologically unrelated wells.

- **Feature-augmented similarity search**  
  KNN neighbor selection uses a feature space containing:
  - monthly water-surface elevation (`WSE`),
  - one-month lag (`LAG_1`),
  - two-month lag (`LAG_2`),
  - rolling mean,
  - rolling standard deviation,
  - month and year information.

- **Transparent donor-well restriction**  
  The default example uses a `GROUP` column derived from the Public Land Survey System (PLSS)-style station identifier prefix, but users can replace this with aquifer codes, basin identifiers, management zones, coordinate-based neighborhoods, or correlation-derived clusters.

- **Station-level and group-level diagnostics**  
  The workflow automatically creates time-series plots distinguishing observed and imputed months.

- **Full-network validation script**  
  A separate validation script performs masked-gap experiments across all eligible wells and compares GWMonthlyKNN with simple baseline methods.

- **Reproducible outputs**  
  The workflow writes machine-readable CSV files and figures that support quality control, review, and reuse.

## Repository Contents

The repository is organized around two main Python scripts.

| File / Folder | Purpose |
|---|---|
| `GWMonthlyKNN.py` | Main imputation script. Reads grouped monthly groundwater data, imputes missing values with KNN, writes completed data, and creates station/group diagnostic plots. |
| `GWMonthlyKNN_full_network_validation_2026.05.31_v6.py` | Full-network validation script. Performs 12- and 24-month masked-gap tests, calculates performance metrics, compares against baseline methods, and generates validation summaries. |
| `gwl-monthly.grouped.csv` | Main input file expected by the scripts. It must contain at least `MSMT_DATE`, `STATION`, `GROUP`, and `WSE`. |
| `gwl-monthly-imputed-KNN.csv` | Main completed output file generated by `GWMonthlyKNN.py`. |
| `time_series_graphs_KNN/` | Station-level time-series plots created by the main imputation script. |
| `group_time_series_graphs_KNN/` | Group-level multiwell time-series plots created by the main imputation script. |
| `infilled_data_KNN/` | Completed station-level CSV files. |
| `estimated_data_KNN/` | Station-level CSV files containing only the imputed months. |
| `outputs_full_network_validation/` | Output folder generated by the validation script. Contains selected validation windows, station-level metrics, method summaries, win-rate summaries, and optional figures. |

Depending on how the repository is distributed, some output folders may not be included initially. They are automatically created when the scripts are run.

## Input Data Format

The main input file must be named:

```text
gwl-monthly.grouped.csv
```

and must be placed in the same working directory as the script, unless the script path is modified.

At minimum, the file must contain the following columns:

| Column | Required | Description |
|---|---:|---|
| `MSMT_DATE` | Yes | Measurement date. It must be readable by `pandas.to_datetime()`. Monthly records are expected. |
| `STATION` | Yes | Unique well or monitoring-station identifier. |
| `GROUP` | Yes | User-defined donor-well group. KNN imputation is performed separately within each group. |
| `WSE` | Yes | Water-surface elevation or groundwater-level value. Missing values should be represented as blank cells or `NaN`. |

Example:

```csv
MSMT_DATE,STATION,GROUP,WSE
2001-01-01,10N04E27R002M,10N04E27,24.15
2001-02-01,10N04E27R002M,10N04E27,24.03
2001-03-01,10N04E27R002M,10N04E27,
2001-01-01,10N04E27R003M,10N04E27,23.91
```

### Meaning of `GROUP`

The `GROUP` column defines the candidate donor-well pool. In simple terms, a well can borrow information only from wells in the same group.

For the California case study, `GROUP` may be derived from the first nine characters of the station code, corresponding to a PLSS-prefix grouping. This is a practical and transparent starting point, not proof that every well in a group taps the same aquifer layer. Users should refine the group definition when better metadata are available.

Possible alternatives include:

- aquifer codes,
- basin identifiers,
- groundwater management zones,
- screened interval classes,
- coordinate-based neighborhoods,
- correlation-based clusters,
- expert-defined hydrogeologic groups.

## Running the Main Imputation Workflow

Place `gwl-monthly.grouped.csv` in the repository folder and run:

```bash
python GWMonthlyKNN.py
```

For Spyder users, the script is designed to run directly with:

```python
runfile("GWMonthlyKNN.py", wdir="path/to/GWMonthlyKNN")
```

The script will:

1. read `gwl-monthly.grouped.csv`;
2. convert `MSMT_DATE` to datetime format;
3. remove stations with no observed WSE data;
4. trim each station to the period between its first and last observed value;
5. remove single-well groups with no missing values;
6. create month and year features;
7. apply group-wise KNN imputation;
8. write the completed dataset;
9. create station-level plots;
10. create group-level plots;
11. save station-specific completed and estimated-only CSV files.

---

## Main Imputation Output

The most important output is:

```text
gwl-monthly-imputed-KNN.csv
```

This file contains the original observations, imputed values, and final completed time series.

Important columns include:

| Column | Meaning |
|---|---|
| `MSMT_DATE` | Date of the monthly record. |
| `STATION` | Well/station identifier. |
| `GROUP` | Donor-well group. |
| `WSE` | Original observed value. Missing months remain blank/NaN in this column. |
| `WSE_imputed` | KNN-estimated value produced by the imputer. |
| `was_missing` | `True` if the original `WSE` value was missing and was filled by the workflow. |
| `WSE_final` | Final completed value. It equals the observed `WSE` when available and equals `WSE_imputed` when the original value was missing. |

For most practical uses, `WSE_final` is the completed groundwater-level series.

## Diagnostic Plots

### Station-level plots

Folder:

```text
time_series_graphs_KNN/
```

Each plot shows the completed monthly time series for one station.

- Observed values are plotted as observed data points.
- Imputed values are highlighted separately.
- Line colors indicate whether a segment connects observed or imputed values.

These figures are useful for visually checking whether imputed values follow the station’s expected seasonal and long-term behavior.

### Group-level plots

Folder:

```text
group_time_series_graphs_KNN/
```

Each plot shows all stations within a group. These plots are useful for checking whether a group behaves coherently and whether imputed values are consistent with neighboring wells.

## Running the Full-Network Validation Script

The repository also includes a full-network validation script:

```text
GWMonthlyKNN_full_network_validation_2026.05.31_v6.py
```

Run it with:

```bash
python GWMonthlyKNN_full_network_validation_2026.05.31_v6.py
```

or in Spyder:

```python
runfile("GWMonthlyKNN_full_network_validation_2026.05.31_v6.py", wdir="path/to/GWMonthlyKNN")
```

The validation script was added to address a common reviewer and practitioner question:

> Does the workflow perform well only for a few selected examples, or does it remain useful across the full monitoring network?

The script evaluates all eligible stations for which valid observed 12- and/or 24-month windows exist.

---

## Validation Design

The validation script performs blocked masked-gap experiments.

In plain language:

1. Find a station with a fully observed 12-month or 24-month block.
2. Temporarily hide that block.
3. Run the imputation workflow as if those values were missing.
4. Compare the imputed values with the true observed values.
5. Repeat this across all eligible stations.
6. Summarize the performance.

This is more realistic than randomly deleting isolated data points because operational groundwater gaps often occur as contiguous blocks caused by outages, maintenance, access limitations, or quality-control exclusions.

### Tested gap lengths

By default:

```python
GAP_LENGTHS = [12, 24]
```

This means that the validation uses 12-month and 24-month artificial outages.

### Window selection rule

By default:

```python
WINDOW_SELECTION = "max_std"
```

This selects the eligible window with the highest standard deviation in WSE. This is intentionally conservative because high-variability periods are more difficult to impute than flat periods.

## Validation Methods Compared

The validation script compares GWMonthlyKNN against practical baseline approaches.

| Method | Description |
|---|---|
| `GWMonthlyKNN` | Feature-augmented group-wise KNN imputation. |
| `baseline_linear_interp` | Time interpolation within the target station. |
| `baseline_seasonal_climatology` | Month-of-year average from the target station. |
| `baseline_neighbor_regression` | Simple regression from the most correlated donor well in the same group, when feasible. |

The baseline methods are intentionally simple. Their purpose is not to represent every possible imputation method, but to provide practical reference points for judging whether GWMonthlyKNN adds value beyond common alternatives.

---

## Validation Metrics

The validation script reports both correlation-based and magnitude-based metrics.

| Metric | Meaning |
|---|---|
| `R` | Pearson correlation between true and imputed values. Measures similarity in shape/timing. |
| `R2` | Squared correlation. Measures explained association, but not absolute error. |
| `RMSE` | Root mean squared error. Penalizes large errors strongly. |
| `MAE` | Mean absolute error. Average absolute error magnitude. |
| `Bias` | Mean signed error, calculated as prediction minus observation. Positive values indicate overestimation; negative values indicate underestimation. |
| `NSE` | Nash-Sutcliffe efficiency. Common hydrologic performance metric comparing predictions to the observed mean. |

Using only correlation can be misleading. A method can reproduce the shape of a hydrograph but still be biased high or low. For that reason, the validation script reports magnitude-based metrics such as RMSE, MAE, Bias, and NSE.

## KNN Configuration

In the main script, the default number of neighbors is:

```python
n_neighbors = 5
```

In the validation script, the default variants are:

```python
KNN_VARIANTS = [
    {"name": "KNN_k5_w3_dist", "k": 5, "w": 3, "weights": "distance"},
    {"name": "KNN_k3_w3_dist", "k": 3, "w": 3, "weights": "distance"},
    {"name": "KNN_k8_w3_dist", "k": 8, "w": 3, "weights": "distance"},
]
```

where:

- `k` is the number of nearest neighbors;
- `w` is the rolling-window length in months;
- `weights="distance"` gives more influence to closer neighbors in feature space.

Users can adjust these settings for sensitivity analysis or site-specific calibration.

## Important Methodological Notes

### 1. GWMonthlyKNN is deterministic

The main imputation script does not use random sampling. Given the same input file and parameters, it should produce the same output.

The validation script sets:

```python
RANDOM_SEED = 42
```

although the default validation window selection is deterministic.

### 2. The workflow does not extrapolate beyond the observation window

Missing values are filled only between the first and last observed value of each station. The method does not estimate values before monitoring begins or after monitoring ends.

### 3. Lag and rolling features are computed from the original series

Lag and rolling features are computed before imputation. During long consecutive gaps, these features may become unavailable after the first few missing months. The KNN imputer then relies on the remaining jointly available features, including month/year information and information from other wells in the group.

### 4. Group definition matters

The quality of imputation depends partly on whether the `GROUP` column defines hydrologically meaningful donor pools. If wells in a group behave very differently because of pumping, screened interval differences, stratigraphy, or local management effects, imputation uncertainty may increase.

### 5. The method is a transparent baseline, not a universal optimum

GWMonthlyKNN is designed to be easy to inspect, reproduce, and adapt. It is not intended to replace site-specific hydrogeologic interpretation or high-capacity probabilistic models when detailed covariates and uncertainty quantification are required.

## Recommended Quality-Control Workflow

After running the scripts, users should:

1. Inspect `gwl-monthly-imputed-KNN.csv`.
2. Review station-level plots in `time_series_graphs_KNN/`.
3. Review group-level plots in `group_time_series_graphs_KNN/`.
4. Check whether imputed values are reasonable during long gaps.
5. Identify groups with weak coherence or unusual hydrographs.
6. If necessary, redefine the `GROUP` column.
7. Re-run the main workflow.
8. Run the full-network validation script.
9. Compare performance against baselines using `summary_by_method_gap.csv`.
10. Document all parameter choices and grouping definitions.

## Common Problems and Solutions

### Problem: `FileNotFoundError: gwl-monthly.grouped.csv`

The input file is not in the current working directory. Place the file in the same folder as the script or change `INPUT_CSV` in the validation script.

### Problem: `KeyError: Missing required columns`

The input CSV does not contain one or more required columns. Check that the file includes:

```text
MSMT_DATE, STATION, GROUP, WSE
```

### Problem: Some stations are not processed

Stations with no observed WSE values are removed. Single-well groups with no missing values are also removed because there is nothing to impute.

### Problem: Neighbor regression baseline has missing results

The regression baseline requires a suitable donor well with enough overlapping observations. If no such donor exists, the method is marked as unavailable for that station.

### Problem: Imputed values look unrealistic

Possible causes include:

- group definition is too broad;
- wells in the same group are not hydrologically coherent;
- the target hydrograph is strongly affected by local pumping;
- gaps are long and concurrent across multiple wells;
- there are too few observed values outside the gap.

Possible solutions:

- refine the `GROUP` column;
- split heterogeneous groups;
- merge adjacent coherent groups;
- test alternative KNN settings;
- compare against baseline outputs;
- add external predictors in a future extension.
